## 2024-06-12 - [Optimize Dockerfile layers and image size]
**Learning:** Found that multiple `apt-get` commands in separate `RUN` directives without `--no-install-recommends` and `rm -rf /var/lib/apt/lists/*` caused unnecessary bloat in the Docker image layers. The lack of `--no-cache-dir` on `pip install` further contributed to a larger image size and potentially cached unnecessary files. Docker layer caching can also be broken due to uncombined `apt-get update` and `apt-get install`.
**Action:** Next time, always consolidate `apt-get update` with `apt-get install` commands into a single `RUN` layer to maximize caching efficiency. Always append `--no-install-recommends` for debian based images to avoid installing optional dependencies not explicitly requested. Conclude all `apt-get install` operations with `rm -rf /var/lib/apt/lists/*` in the same RUN layer to remove fetched package lists directly, saving noticeable space. Finally, always append `--no-cache-dir` to `pip install` directives to prevent accumulating tarball caches locally during CI builds unless explicitly caching in a multi-stage Dockerfile environment (though the pip cache won't be in the final stage here, it's a good generic standard).
## 2024-06-18 - [Optimize Docker image size and cold start time]
**Learning:** Found two interesting performance optimizations in Dockerfile construction:
1. Deleting `__pycache__` to save disk space is an optimization that surprisingly backfires. It trades a tiny image size reduction for slower container startup times (cold starts), as Python is forced to recompile `.py` files into bytecode on the fly every time the container starts.
2. The final stage was using `python:3.15.0b2-bookworm` (~1.5GB) instead of `python:3.15.0b2-slim-bookworm` (~189MB). This massive size difference significantly impacts pull times and extraction times during deployment.
**Action:** Next time, preserve `__pycache__` directories in Python application containers to prioritize fast startup times over marginal disk space savings. Always verify that final multi-stage build images are using `-slim` or equivalent variants to minimize the final deployment footprint.
## 2024-06-25 - [Optimize Python cold start time in multi-stage Docker builds]
**Learning:** Found a major performance bottleneck where a multi-stage Dockerfile used a `build_env` stage with `debian:bookworm` (Python 3.11 default) to `pip install` dependencies, but copied them to a final stage using `python:3.15.0b2-slim-bookworm`. This version mismatch causes the generated `.pyc` (bytecode) files to be completely useless. When the final container runs, Python 3.15 is forced to recompile all `.py` files into new `.pyc` files on the fly, adding significant overhead to the cold start time.
**Action:** Next time, always ensure the exact same Python version (and preferably the same base image tag) is used in both the build environment stage and the final execution stage of a Dockerfile. This guarantees that bytecode generated during dependency installation is fully compatible and instantly usable at runtime, optimizing cold start times.
